# 需求分析报告

## 概述

### 基本要求

本项目旨在开发一个面向在线图书销售的电子商务 Web 应用系统，支持三类用户角色（用户、销售人员、管理员）协同工作，实现从商品浏览、购物车、订单管理到销售分析、用户画像、推荐系统的完整业务流程，并集成大数据采集与分析能力。

### 基本内容

**Step 1：业务场景分析**

传统实体书店存在以下痛点：库存管理依赖人工记录，效率低下且易出错；销售数据无法实时统计，经营者难以及时掌握经营状况；用户购买偏好无法有效记录和利用，缺乏个性化推荐；顾客需要到店选购，受时间和空间限制。

在线书店系统可以解决上述问题：通过信息化手段实现图书的在线展示与交易，自动记录库存变化和销售数据，基于用户行为日志进行大数据分析，提供个性化推荐服务，同时支持销售人员和管理员的分工协作。

**Step 2：可能的解决方案**

方案一：采用 Spring Boot + Vue.js 的前后端分离架构，适合大型项目但开发周期较长。
方案二：采用 JSP + Servlet + JDBC 的传统 Java Web 架构，开发效率高，适合课程设计周期要求。
对比后选择**方案二**，原因：技术栈与课程教学内容高度契合，团队对 Java EE 技术栈熟悉，开发效率高。

**Step 3：可行性分析**

- 技术可行性：JSP + Servlet + MySQL 技术栈成熟稳定，社区支持丰富，学习曲线平缓
- 经济可行性：使用开源技术，零授权成本，部署于华为云 ECS，成本可控
- 操作可行性：B/S 架构，用户通过浏览器即可访问，无需安装客户端

**Step 4：实施方案与技术指标**

| 阶段 | 内容 | 时间 |
|------|------|------|
| 需求分析 | 梳理功能需求，设计数据库 | 第1周 |
| 核心开发 | 用户端注册/登录/浏览/购物车/下单 | 第2-3周 |
| 后台开发 | 销售人员/管理员后台 + 权限系统 | 第3-4周 |
| 增强功能 | 日志追踪/邮件发送/销售分析/用户画像/推荐系统 | 第5-6周 |
| 部署测试 | 华为云部署，功能测试，撰写报告 | 第7周 |

**技术指标**：支持 3 种用户角色权限隔离、IP+时长+分类的行为日志采集、日/周/月销售趋势图表与简单预测、基于协同过滤的图书推荐、10 秒内 60 次请求的爬虫频率限制。

# 系统设计报告

## 概述

### 整体框架设计

系统采用经典的 MVC 三层架构：

- **表现层（View）**：JSP 页面 + Bootstrap 3 + Chart.js/ECharts 数据可视化
- **控制层（Controller）**：Servlet 处理 HTTP 请求，Filter 实现编码/登录验证/频率限制
- **模型层（Model）**：Bean 实体类 + DAO 数据访问层 + JDBC 直连 MySQL

```
浏览器 → Filter（编码/登录/频率） → Servlet → DAO → MySQL
                                        ↓
                                   JSP（渲染响应）
```

### 模块划分

**1. 用户模块**
- 注册（含邮箱选填）、登录（记录IP）、注销
- 个人信息查看/编辑/账号注销
- 登录状态 AJAX 检测

**2. 图书浏览与购物模块**
- 分类浏览、关键词搜索、详情查看（记录浏览时长+分类）
- 购物车（加入/修改数量/删除）
- 订单提交（库存扣减）、邮件确认（QQ SMTP）
- 「看了又买」关联推荐 + 「购买此书的也买了」协同过滤推荐

**3. 销售管理模块（销售人员）**
- 图书管理：列表查看、价格/库存修改
- 分类管理：添加/删除图书类别
- 订单管理：订单列表、订单处理（发货）
- 销售分析：销售监控仪表盘、销售报表（日/周/月趋势+预测+排行）
- 用户分析：用户画像、行为日志

**4. 管理员模块**
- 人员管理：销售人员账号管理（添加/删除/密码重置）、用户管理、工作人员日志
- 除人员管理外，拥有销售人员全部权限
- 异常监控：今日销量对比告警、零销售分类、低库存热销预警

**5. 日志与监控模块**
- s_operation_log 表统一记录所有行为（BROWSE/PURCHASE/EMAIL/LOGIN/ADMIN_OP/CRAWLER）
- IP 地址采集（X-Forwarded-For / X-Real-IP / RemoteAddr）
- 浏览时长前端 JS 计时 + sendBeacon 回传
- 反爬虫：RateLimitFilter，10秒60次限流，拦截写入日志

### 数据库设计

共 7 张业务表 + 2 张视图：

| 表名 | 说明 | 核心字段 |
|------|------|---------|
| s_admin | 后台用户 | id, userName, passWord, role(admin/sales) |
| s_user | 前台用户 | userId, userName, email, address, enabled |
| s_book | 图书 | bookId, catalogId, price, stock |
| s_catalog | 分类 | catalogId, catalogName |
| s_order | 订单 | orderId, userId, orderStatus(1待处理/2已发货/3已完成) |
| s_orderitem | 订单明细 | bookId, orderId, quantity |
| s_operation_log | 行为日志 | userId, logType, targetId, ipAddress, duration, catalogId |
| s_uploadimg | 图片存储 | imgId, imgSrc |
| s_user_profile | 用户画像缓存 | userId, totalSpent, favCatalogId |

### 大数据方案

**数据采集**：
- 用户登录信息（时间、IP地址）
- 浏览行为（商品类别catalogId、停留时长duration秒）
- 购买记录（商品类别、日期、单价、数量）
- 销售人员/管理员登录及操作日志

**数据分析**：
- 用户画像：地域分布、性别/年龄饼图、购买力分层（低<100/中100-300/高>300）、分类偏好
- 销售排行：TOP 20 销量/金额排行榜，横向条形图
- 趋势预测：日/周/月折线图 + 7日/4周/3月移动平均预测虚线
- 异常判别：今日销量vs前7天日均（<30%告警）、零销售分类、低库存热销预警

**推荐系统**：
- 关联规则推荐："浏览过此书的人也买了"（s_operation_log BROWSE→PURCHASE 关联）
- 协同过滤推荐："购买此书的也买了"（s_orderitem 同单共现）

### 风险评估与预案

| 风险 | 影响 | 预案 |
|------|------|------|
| SQL 注入 | 数据泄露 | 所有查询使用参数化 PreparedStatement（少量动态表名/排序字段使用白名单校验）|
| 权限绕过 | 越权操作 | FilterLogin 双重校验（JSP标签隐藏 + Filter 服务端拦截） |
| 爬虫攻击 | 服务器压力 | RateLimitFilter 基于 IP 的频率限制 |
| 邮件发送失败 | 用户体验下降 | try-catch 包裹，失败记录日志但不阻塞主流程 |
| 数据库连接泄露 | 服务不可用 | DbUtil 在 finally 块统一关闭连接 |

### 核心实现说明

**1. 权限控制（FilterLogin）**
- 未登录访问 /jsp/admin/* → 重定向到登录页
- admin 角色：所有功能
- sales 角色：禁止访问 AdminManageServlet、UserManageServlet、adminManage/*、userManage/* JSP、统计报表 stat 操作

**2. 行为日志（OperationLogUtil + s_operation_log）**
- BROWSE：bookdetailed.java 记录，含 IP、catalogId、duration
- PURCHASE：OrderSubServlet 记录，含 IP、金额
- LOGIN：UserServlet/LoginServlet 记录，含 IP、角色
- EMAIL：EmailUtil 记录，含发送状态
- ADMIN_OP：AdminManageServlet/BookManageServlet/CatalogServlet 记录，含操作内容、IP
- CRAWLER：RateLimitFilter 记录，含请求数

**3. 邮件发送（EmailUtil + JavaMail 1.6.2）**
- 用户下单后，若 user.email 不为空，通过 QQ SMTP（SSL 465）发送确认邮件
- 发送结果写入 s_operation_log

**4. 销售报表（OrderManageServlet + Chart.js）**
- 月度销售条形图
- 日/周/月趋势折线图 + 移动平均预测虚线
- 类别销量条形图
- 商品销量排行 TOP20
- 订单状态汇总表 + 库存汇总表

**5. 推荐系统**
- "看了又买"：在同一本书上浏览过的用户购买的图书，按频次 TOP5
- "购买此书的也买了"：同一订单中共现的图书，按频次 TOP5

**6. 反爬虫（RateLimitFilter）**
- 每个 IP 维护 10 秒滑动窗口，超过 60 次返回 HTTP 429
- 拦截记录写入 s_operation_log（logType=CRAWLER）

### 遇到的问题及解决方案

**案例1**

【问题描述】数据库 s_operation_log 表增加了 ipAddress、duration、catalogId 三个字段，但部署后执行 ALTER TABLE 时报错 "Error 1060: Duplicate column name"，因为 SQL 文件中同时存在 CREATE TABLE（已含新字段）和 ALTER TABLE（重复添加）。

【AI 的建议】AI 分析发现 CREATE TABLE 已经定义了新字段，建议删除冗余的 ALTER TABLE 语句，保留 UPDATE 数据初始化语句。

【实际采纳情况】完全采纳

【最终解决方案】在 bookshop.sql 中将 ALTER TABLE 添加字段的语句注释掉，只保留 CREATE TABLE 中的字段定义和 UPDATE 数据种子语句，重新导入数据库后正常。

【反思】数据库脚本需要区分"全新安装"和"增量升级"两种场景。CREATE TABLE 适用于全新安装，ALTER TABLE 适用于已有数据库升级。两者的字段定义应保持互斥，避免重复。

**案例2**

【问题描述】销售报表页面的日趋势图和周趋势图数据为空，但月趋势图只显示到 2025 年 12 月就没了。排查发现是 bookshop.sql 种子数据的订单日期全部在 2025 年，而日/周查询用了 `date_sub(curdate(), interval 30 day)` 只查最近 30 天。

【AI 的建议】AI 建议将种子数据中的订单日期全部更新为 2026 年 1 月至 5 月，使日/周/月三个维度的查询都能覆盖到数据。同时建议生成更多的种子数据条目使报表"好看"。

【实际采纳情况】完全采纳

【最终解决方案】将 25 条订单的日期从 2025 年重新分配为 2026 年 1 月至 5 月，覆盖 5 个月份，并增加了 3 条新订单使 5 月有 8 条。同时添加了 28 条 s_operation_log 种子数据（登录、浏览、购买、邮件、工作人员操作）。

【反思】种子数据的时效性很重要。对于有时间维度查询的报表功能，数据日期需要覆盖当前时间窗口，否则看起来像功能有问题。

**案例3**

【问题描述】管理员后台侧边栏的"用户分析"菜单前面出现一条异常的白色竖线，其他菜单正常，但反复检查 HTML 结构没有发现问题。

【AI 的建议】AI 分析是 CSS 问题。left.css 中 `.left .list h3::before` 默认规则创建了一个 4px 白色竖条，而 nth-child(1-3) 用 emoji 图标覆盖了默认规则。第 4 个菜单"用户分析"没有对应的 nth-child(4) 规则，回退到了默认的白条。

【实际采纳情况】完全采纳

【最终解决方案】将默认 `::before` 改为小灰点 `content: '●'` 避免断裂的竖条效果，并为 nth-child(4) 和 nth-child(5) 添加了图标规则。同时修复了左侧菜单 HTML 中 `<li class="list">` 丢失导致的结构错误。

【反思】CSS 的 nth-child 选择器和回退机制在动态渲染不同数量的菜单项时需要特别注意。应对所有可能的菜单项数量（管理员 4 个、销售人员 4 个）分别定义 nth-child 规则，并用安全的默认回退样式兜底。

## 测试方法

本系统采用**黑盒测试**和**功能测试**相结合的方法，从用户视角出发，不关注内部代码逻辑，只验证输入输出是否符合预期。
